# VL 模型健康场景优化分析

## 1. 优化背景

PHA 集成 VL 模型（qwen3-vl-235b）实现图片健康分析后，在真实链路测试中暴露出多个问题。本文档记录从测试发现到根因分析到优化措施的完整链路，涵盖模型适配层优化、Prompt 工程优化、数据预处理策略优化三大方向。

## 2. 模型适配层优化

### 2.1 问题：VL 多轮对话 content=null 导致空回复

| 项目 | 内容 |
|------|------|
| 发现方式 | vl-matrix-test T3/T5/T6 场景 FAIL |
| 根因 | vLLM 引擎要求 assistant message 的 content 字段非 null/空串，否则返回 0 token |
| 影响范围 | 所有经过工具调用的多轮对话（agent 第二轮开始 content 常为 null） |
| 修复方案 | 在 VL 适配层将 null/空串替换为非空 placeholder |
| 修复 commit | `efff1f3` fix: use non-empty placeholder for assistant content in VL tool_calls |
| 验证 | T3/T5/T6 场景从 FAIL 变为 OK |

**优化前后对比：**
```
修复前: T1=OK T2=OK T3=FAIL T4=OK T5=FAIL T6=FAIL T7=OK -> 通过率 57%
修复后: T1=OK T2=OK T3=OK  T4=OK T5=OK  T6=OK  T7=OK -> 通过率 100%
```

### 2.2 问题：VL 流式传输内容缓冲

| 项目 | 内容 |
|------|------|
| 发现方式 | 用户上传图片后前端无实时反馈 |
| 根因 | TransformStream 缓冲机制导致 VL 模型的 content chunk 积攒后才推送 |
| 修复方案 | 改为每个 chunk 立即推送给 SSE 流 |
| 修复 commit | `f6cebc1` fix: stream VL content chunks immediately instead of buffering |

### 2.3 问题：VL adapter fetch 方案三次迭代

| 迭代 | 方案 | 问题 | commit |
|------|------|------|--------|
| v1 | globalThis.fetch wrap | 与 Vercel AI SDK 的 fetch 冲突 | `a7ec241` |
| v2 | custom OpenAI fetch | 绕过 SDK 内部 fetch 管理 | `8a53165` |
| v3 | onPayload hook | 最小侵入，在 payload 层面修正 content 字段 | `f2bc1a5` |

**最终选择 v3（onPayload hook）的理由：**
- 不修改 fetch 全局行为，无副作用
- 只在发送 payload 前修正 assistant.content 字段
- 与 Vercel AI SDK 的 stream/non-stream 路径均兼容

### 2.4 问题：VL 模型文本 tag 替代标准 tool_call

| 项目 | 内容 |
|------|------|
| 发现方式 | VL 模型返回文本 tag 而非标准 tool_call JSON |
| 根因 | 部分 VL 模型（尤其量化版本）的 function calling 能力退化，降级为文本 tag |
| 修复方案 | 在 VL adapter 中增加 ECHO 分支，后处理解析文本 tag 并合成标准 tool_call 结构 |
| 修复 commit | `b29dbc0` fix: VL adapter ECHO branch -- synthesize extra tool calls from text tags |

## 3. Prompt 工程优化

### 3.1 问题：上下文长度退化

**测试数据（test-vl-context-length.sh）：**

| prompt 大小 | 工具调用成功率 | 问题 |
|------------|--------------|------|
| < 5KB | ~100% | 无退化 |
| 5-8KB | ~70% | WRONG_ARGS 开始出现 |
| 8-20KB | < 50% | TEXT_REPLY / EMPTY 频繁 |

**优化措施：**

1. **Skill 按需加载**：移除 image-routing skill 的预注入，改为模型主动调用 get_skill 按需加载
   - commit: `8fd8bec` fix: remove image-routing skill pre-injection, use same flow as text
   - 效果：system prompt 减少 ~3KB

2. **get_skill 返回体积精简**：删除 get_skill 工具返回的重复 content 字段
   - commit: `d572324` fix: remove duplicated content field from get_skill response
   - 效果：单次 skill 加载从 ~30KB 降至 ~15KB

3. **通用图片流程**：移除 diet-specific 硬编码指令，改为通用图片识别 -> skill 加载流程
   - commit: `16de51f` fix: remove diet-specific instruction from generic image path
   - 效果：减少不必要的 prompt 指令

### 3.2 问题：VL 模型不主动调用 get_skill

| 项目 | 内容 |
|------|------|
| 发现方式 | 多模型评测中发现部分模型收到图片后直接文字回复，不调 get_skill |
| 根因分析 | VL 模型的指令遵循能力弱于纯文本模型，长 system prompt 中的工具调用指令被稀释 |
| 当前状态 | 残留问题，需从 prompt 强化 + 消融实验结论综合解决 |

**消融实验为此提供的优化方向：**
- 条件 B（+页面描述）和条件 D（+描述+知识）下模型的工具调用时机评分更高
- 建议：将核心工具调用指令移至 system prompt 开头或以 few-shot 示例强化

## 4. 数据预处理策略优化

### 4.1 DOC vs RAW 实验结论

基于 run_doc_vs_raw.py 的对比实验，得出数据预处理策略：

| 策略 | DOC（结构化中文） | RAW（SDK 原始 JSON） |
|------|------------------|---------------------|
| 异常识别 | 更好（+1.2 分） | 较差 |
| 单位换算 | 较差（-1.0 分） | 更好 |
| 工具调用完整性 | 更好（3-tool pipeline） | 较差（单 tool） |
| 整体推荐 | 端侧场景推荐 | - |

**优化建议：** 端侧数据应在传入模型前做结构化预处理（terminal_data -> 中文语义描述），可显著提升异常识别能力和工具调用完整性。

### 4.2 消融实验对知识注入策略的指导

| 注入方式 | 增益 | 副作用 | 建议 |
|----------|------|--------|------|
| 页面描述（description.json） | 视觉识别 +0.5~1.0 | 增加上下文长度 | 按 currentPage 精准注入 |
| 健康知识（knowledge.json） | 安全声明 +1.0~1.5 | 知识覆盖端侧数据倾向 | 控制注入长度，标注来源 |
| 描述 + 知识同时注入 | 综合最优 | 小模型上下文退化 | 大模型（397B+）推荐 |

**优化实施方案：**
- 通过 keyword_dict.json 做 currentPage -> 知识条目的精准映射，避免全量注入
- 注入时明确标注来源区分
- 模型选择：397B 以上使用条件 D（全量注入），122B 以下使用条件 A（避免退化）

## 5. 模型稳定性优化

### 5.1 GLM-4.6V crash 问题分析

从消融实验评分数据提取 GLM-4.6V 的 crash pattern：

| Case | 现象 | 输出示例 |
|------|------|---------|
| row007 | crash truncation | 调用 get_skill 两次后无输出 |
| row010 | garbled output | token 错误乱码 |
| row017 | 严重 crash | 随机数字和字母，无有效内容 |
| row021-032 | 批量 crash | 多个 case 连续 crash |

**根因：** GLM-4.6V 通过 OpenRouter 中转时，长 system prompt + 图片 base64 的组合触发了模型的不稳定区域。

**优化建议：**
- 生产环境不使用 GLM-4.6V
- 如必须使用，增加 response 有效性检测 + 自动重试

### 5.2 幻觉控制优化

从评分数据提取高频幻觉模式：

| 幻觉类型 | 示例 | 涉及模型 | 防控措施 |
|----------|------|---------|---------|
| 数值编造 | 睡眠得分 95（实际 85） | Qwen3.5-122B | 端侧数据 grounding |
| 趋势编造 | 编造周趋势数据 | Qwen3.5-397B | 限制模型推断范围 |
| 数据混淆 | 云数据混入端侧回复未标来源 | Qwen3-235B | 强制标注数据来源 |
| 跨字段错误 | 把步数数据填入心率字段 | Qwen3.5-122B (row004) | Structured output 约束 |

## 7. 测试脚本与优化的闭环关系

```
测试发现 -------> 根因分析 -------> 优化实施 -------> 回归验证
   ^                                                    |
   +----------------------------------------------------+

具体闭环示例:

vl-matrix-test T6=FAIL
  -> 根因: vLLM 不接受 content=null
    -> 修复: efff1f3 non-empty placeholder
      -> 回归: T6=OK

test-vl-context-length F=EMPTY
  -> 根因: 20KB system prompt 超出模型有效范围
    -> 修复: 8fd8bec 移除 skill 预注入 + d572324 精简返回
      -> 回归: 有效 prompt 控制在 5KB 内

run_onapp_eval dim9=2-3 (所有模型)
  -> 根因: 异常 case 下模型不主动调云侧工具
    -> 进行中: prompt 强化 + few-shot 示例
      -> 待验证

run_doc_vs_raw DOC 异常识别 +1.2
  -> 根因: 结构化中文数据帮助模型理解异常语义
    -> 建议: 端侧数据预处理管线
      -> 待实施
```

每个优化措施都可追溯到具体的测试脚本和测试数据，形成「测试驱动优化」的闭环。
